# CLAUDE.md
# Strict Execution Contract

## 最高原则
- 先确认事实，再执行操作，再修改代码，再验证结果。
- 不允许跳步、不允许猜测、不允许凭经验臆断。
- 没有文件证据、日志证据或实际输出，就不能下结论。
- 未验证完成前，不允许宣布完成。
- 任务的目标不是“看起来对”，而是“可复现地正确”。

---

## 绝对优先级
1. 当前仓库文件与目录结构
2. package.json / 配置文件 / 终端日志
3. 实际命令执行结果
4. 页面或功能验证结果
5. 才能进行总结或汇报

---

## 通用执行流程
所有任务都必须遵守以下流程，除非当前任务明确不需要其中某一步：

### 1. 识别
- 先读取项目结构。
- 先读取 package.json。
- 先确认当前技术栈、可用脚本、入口文件。
- 先确认当前任务属于：查看 / 启动 / 调试 / 修改 / 构建 / 验证 中的哪一种。

### 2. 计划
- 先列出准备执行的操作。
- 先确认会动哪些文件、会跑哪些命令、预期会看到什么结果。
- 计划必须最小化，优先选择最短路径。

### 3. 执行
- 严格按计划执行。
- 优先使用仓库内已有脚本。
- 不要凭感觉额外加命令。
- 不要在没有证据前改多个无关文件。

### 4. 验证
- 每次修改后都要验证。
- 验证优先级：终端日志 > 构建结果 > 页面可访问性 > 功能可用性。
- 验证失败就回到日志，不要继续盲改。

### 5. 汇报
- 只汇报当前真实状态。
- 成功就说明成功依据。
- 失败就说明失败原因、卡点和下一步。

---

## 项目识别规则
在进行任何代码修改之前，必须先完成：

1. 读取 `package.json`
2. 列出可用 scripts
3. 判断项目类型
4. 判断最合适的启动方式

### 默认判断原则
- Vite：优先 `npm run dev`
- Next.js：优先 `npm run dev`
- React CRA：优先 `npm start`
- Node 服务：优先仓库内定义的 dev/start 脚本
- 预览模式：只在构建成功且明确需要预览时使用

### 禁止行为
- 禁止不看 `package.json` 直接猜命令。
- 禁止把构建命令当成开发启动命令。
- 禁止为了“试试看”乱跑多个命令。
- 禁止在未确认框架前先改代码。

---

## 启动流程硬规则
在修改代码之前，必须先确认项目是否能正常启动。

### 必须做
- 读取启动脚本。
- 启动 dev server。
- 等待成功输出。
- 识别本地访问地址。
- 确认端口是否占用。
- 确认是否有 runtime error。

### 必须输出
- 使用了什么命令
- 启动到哪一步
- 本地访问地址
- 是否成功
- 成功或失败的依据

### 禁止
- 不看日志就判断启动成功。
- 没有成功输出就提前结束。
- 看到报错不处理还继续下一步。
- 启动失败后直接改代码，不先读日志。

---

## 调试规则
当出现错误时，必须按顺序处理：

1. 读取完整终端日志
2. 找到第一条真实错误
3. 找到错误对应的文件、行号或模块
4. 基于日志判断原因
5. 只提出一个最优修复方案
6. 做最小修改
7. 重新验证

### 调试原则
- 先证据，后结论。
- 先定位，后修改。
- 先单点修复，后扩展。
- 一次只解决一个主要问题。
- 不允许同时尝试多个不相关修复。

### 禁止
- 纯猜测修 bug。
- 连续盲改多个文件。
- 看到 warning 就忽略。
- 没有定位就大改架构。
- 用“可能是”代替实际诊断。

---

## 代码修改规则
- 只改与当前任务直接相关的文件。
- 优先最小改动。
- 不要顺手重构无关代码。
- 不要为了“更规范”破坏现有稳定性。
- 如果必须新增文件，先确认它是否真的必要。
- 修改后必须重新验证，不能只凭静态判断。

### 优先级
- 修复当前问题 > 保持最小改动 > 美观优化 > 结构重构

---

## 验证规则
每次修改后必须验证，验证方式按场景选择：

### 开发启动类任务
- 查看终端是否有成功启动信息
- 确认访问地址
- 打开页面确认是否可访问

### 修改功能类任务
- 重新运行相关命令
- 查看是否有报错
- 核对修改是否生效
- 核对是否引入新错误

### 构建类任务
- 运行 build
- 检查是否通过
- 读取构建日志
- 记录失败点

### 验证失败后的处理
- 回到日志
- 找出新错误
- 只修一个最关键问题
- 再次验证

---

## 不确定情况处理
遇到以下情况必须停止并汇报事实：

- `package.json` 缺失或损坏
- scripts 不明确
- 启动脚本不存在
- 端口冲突
- 依赖缺失
- 环境变量缺失
- 日志无法直接定位问题
- 多个错误同时存在且无法判断主因

### 处理方式
1. 说明当前事实
2. 说明卡在哪一步
3. 说明还缺什么信息
4. 再决定下一步

---

## 任务分流规则
不同任务采用不同策略，但都必须遵守“先证据后动作”。

### 查看类
- 先读文件
- 再总结
- 不做无关修改

### 启动类
- 先识别脚本
- 再启动
- 再确认地址
- 再汇报状态

### 调试类
- 先读日志
- 再定位
- 再最小修复
- 再验证

### 改动类
- 先确认现有结构
- 再做最小改动
- 再验证结果

### 构建/发布类
- 先确认构建命令
- 再构建
- 再检查产物和日志
- 再汇报

## 终止条件
只有同时满足以下条件，才能认为当前任务完成：

- 目标已经实现，或者问题已明确定位
- 相关修改已经完成
- 已重新验证
- 没有新的关键错误
- 能明确说明当前状态或访问地址

---

## 额外约束
- 不允许在未验证前使用“已完成”这种表述。
- 不允许用模糊话术代替实际结果。
- 不允许把推断说成事实。
- 不允许忽略日志中的关键错误。
- 不允许为了节省步骤而省略验证。
- 不允许在没有证据时给出确定性结论。

---

## 核心目标
把模型行为锁定为：

- 先读事实
- 再做计划
- 再执行命令
- 再读取日志
- 再最小修复
- 再验证结果
- 最后才汇报结论
